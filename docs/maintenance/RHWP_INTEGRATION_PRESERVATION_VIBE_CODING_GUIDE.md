# RHWP Integration Preservation Framework 바이브 코딩 가이드

Project:
- `RHWP Integration Preservation Framework`
- `RHWP 엔진 통합 보존 프레임워크`

## 목적

이 문서는 `RHWP 엔진 통합 보존 프레임워크`에서 사용하는 개발 방식을 이 프로젝트 현실에 맞게 정의한다.

이 프로젝트의 개발 방식은 전통적인 의미의 엄격한 AI 페어 프로그래밍만으로 설명되지 않는다.
오히려 실제 작업 방식은 `바이브 코딩에 가깝다`.

다만 이 프로젝트의 바이브 코딩은
아무 생각 없이 AI 출력을 받아들이는 무통제 방식이 아니라,
빠른 실험과 빠른 구현을 허용하되
중요한 경계와 검증만은 놓치지 않는 방식이다.

한 줄로 정의하면:

`검증과 경계를 갖춘 실전형 바이브 코딩`

## 이 프로젝트에서의 바이브 코딩 정의

이 프로젝트에서 바이브 코딩은 다음을 의미한다.

- 사람은 세부 구현을 처음부터 끝까지 엄격하게 설계하지 않아도 된다.
- AI는 빠르게 구조, 코드, 실험안, 수정안을 제시하고 실제 구현까지 밀어준다.
- 사람은 구현 전부를 line-by-line으로 설계하지는 않지만,
  방향, 취소선, 품질 기준, 최종 수용 여부는 계속 쥐고 있어야 한다.
- 완벽한 선계획보다
  빠른 시도 -> 확인 -> 수정 -> 재시도
  흐름을 허용한다.

즉 이 프로젝트는
`계획 없는 난사형 바이브 코딩`은 아니고,
`문서와 검증을 최소 안전장치로 둔 고속 바이브 코딩`
에 가깝다.

## 왜 AI 페어 프로그래밍만으로는 설명이 안 되는가

이 프로젝트는 실제로 아래 특징이 있다.

- 사용자가 구현 방향을 매우 빠르게 바꾼다.
- UX 피드백이 실시간으로 반영된다.
- 먼저 만들어보고 써본 다음, 다시 구조를 정리하는 일이 많다.
- 문서도 중요하지만, 문서보다 먼저 실사용 감각과 체감 성능을 확인하는 경우가 많다.
- “이론적으로 더 좋은 구조”보다 “지금 체감상 더 좋은 동작”이 우선될 때가 많다.

이건 전형적인 문서 우선형 개발이라기보다,
실전형 바이브 코딩에 더 가깝다.

그래서 이 프로젝트의 정체성은 이렇게 보는 게 맞다.

- 기본 성격: 바이브 코딩
- 안전장치: 검증, 문서, 경계 통제
- 목표: 감각적으로 빠르되, 나중에 망하지 않게 유지

## 이 프로젝트가 지향하는 방식

이 프로젝트는 아래 둘 중 하나가 아니다.

### 1. 무통제 바이브 코딩

- AI가 만든 걸 거의 안 읽고 붙임
- 되면 장땡
- 나중에 왜 되는지 모름
- 엔진 경계도 흐려짐

이건 아니다.

### 2. 과도하게 무거운 정통 프로세스 개발

- 모든 변경 전 상세 설계 필수
- 모든 결정이 문서 선행
- 속도가 지나치게 느려짐
- 실사용 피드백 반영이 늦음

이것도 현재 프로젝트 성격과는 안 맞는다.

### 3. 이 프로젝트의 실제 방식

- 일단 빠르게 만든다.
- 바로 실행해본다.
- 체감으로 이상한 점을 잡는다.
- AI가 고친다.
- 문서화는 흐름을 잃지 않을 정도로 유지한다.
- 중요한 분기점에서만 구조와 검증을 강하게 잡는다.

즉:

`빠르게 만들고, 바로 보고, 바로 고치되, 중요한 곳은 절대 대충 넘기지 않는다`

## 프로젝트용 개발 철학

## 1. 속도는 장점이다

이 프로젝트에서는 속도가 미덕이다.

빠르게 시도하고,
빠르게 앱을 실행하고,
빠르게 체감 피드백을 받고,
빠르게 다시 수정하는 흐름 자체가 생산성의 핵심이다.

AI는 이 속도를 극대화하는 도구다.

## 2. 체감 UX가 문서보다 먼저일 수 있다

이 프로젝트에서는 특히 아래 영역에서 체감이 매우 중요하다.

- 인쇄 UX
- PDF 내보내기 UX
- 진행률 표시
- ETA 신뢰성
- 내부 PDF 뷰어의 이질감
- 링크 드롭의 즉시성

즉, 먼저 만져보고 이상하면 고치는 흐름이 자연스럽다.

문서는 이 흐름을 보조하고 정리하는 역할을 한다.

## 3. 그래도 절대 대충 넘기면 안 되는 경계가 있다

바이브 코딩 중심 프로젝트라도,
다음은 반드시 통제해야 한다.

- RHWP 코어와 BBDG 기능의 경계
- 현재 업그레이드된 기능의 보존
- UI/UX 회귀 여부
- 엔진 업데이트 수용성
- 오류 검증
- 성능 검증

즉, 구현 과정은 자유롭게 흔들릴 수 있어도
결과물의 안전 경계는 흔들리면 안 된다.

## 이 프로젝트에서의 역할 분담

## 사람

사람은 이 프로젝트에서 다음을 맡는다.

- 감각적 방향 제시
- UX 좋고 나쁨 판단
- 우선순위 결정
- “이건 아닌데?”를 말하는 역할
- 최종 수용 판단
- 큰 분기 결정

사람은 모든 코드를 처음부터 설계하지는 않는다.
하지만 프로젝트의 감각과 기준은 계속 잡고 있어야 한다.

## AI

AI는 이 프로젝트에서 다음을 맡는다.

- 빠른 구현
- 빠른 수정
- 대안 제안
- 리팩터링
- 로그 분석
- 테스트 실행
- 문서 정리
- 반복 작업 처리

AI는 이 프로젝트에서 단순 보조보다 더 큰 역할을 한다.
실제로 구현 엔진에 가깝다.

## 이 프로젝트의 작업 루프

이 프로젝트는 아래 루프로 움직인다.

```text
아이디어/불만/문제 제기
-> AI가 빠르게 구현 또는 수정
-> 바로 실행
-> 체감 확인
-> 이상하면 즉시 수정
-> 충분히 좋아졌으면 문서/구조 정리
-> 중요한 단계에서 검증/보존 체크
```

이 순서는 전통 개발보다 훨씬 유동적이다.
그게 이 프로젝트의 실제 작업 방식이다.

## 계획의 위치

이 프로젝트는 계획을 쓰긴 한다.
하지만 계획이 구현보다 항상 먼저 오는 것은 아니다.

더 정확히는:

- 큰 분기에서는 계획이 먼저 온다.
- 작은 개선은 구현과 실험이 먼저 올 수 있다.
- 계획은 프로젝트를 묶는 역할을 하고,
  바이브 코딩은 실제 속도를 만든다.

즉:

`거친 방향은 문서로 잡고, 실제 이동은 바이브로 한다`

## 이 프로젝트에서 허용되는 것

- AI가 먼저 시안을 만들고 사람이 보고 수정하는 것
- 빠른 프로토타이핑
- UI를 직접 보고 즉시 조정하는 것
- 구조를 조금 미뤄두고 UX부터 맞추는 것
- 먼저 동작하게 만들고 나중에 정리하는 것
- 문서와 구현이 완전히 동시에 가지 않아도 되는 것

## 이 프로젝트에서 금지되는 것

바이브 코딩 중심이라고 해서 아래까지 허용되지는 않는다.

- RHWP 코어를 생각 없이 오염시키는 것
- generated 파일을 수동 수정하는 것
- 기능 회귀를 검증 없이 넘어가는 것
- “되긴 되니까”로 push 하는 것
- 인쇄/PDF/link-drop 같은 핵심 UX를 망가뜨리고도 진행하는 것
- engine boundary를 무시한 대충 구현
- 원인 분석 없이 랜덤 패치 반복

## 검증형 바이브 코딩의 최소 규칙

이 프로젝트에서는 아래 최소 규칙만은 반드시 지킨다.

### 1. 큰 변경 전

- 어떤 층을 건드는지 확인
- RHWP 코어인지, adapter인지, app service인지 구분
- 현재 기능이 무엇이 깨질 수 있는지 생각

### 2. 변경 후

- 실제 실행
- 실제 체감 확인
- 오류 로그 확인
- 기능 유지 여부 확인

### 3. 분기점에서

- 문서 갱신
- 아키텍처 경계 재확인
- 체크리스트 확인
- 필요하면 report 남기기

## 프로젝트 전용 성공 기준

이 프로젝트에서 성공은 “코드가 예쁘다”가 아니다.

성공은 아래 상태다.

- 실제 써보면 좋아졌다
- 현재 BBDG 기능이 유지된다
- 인쇄/PDF/link-drop 흐름이 살아 있다
- RHWP 업데이트를 다시 받을 길이 남아 있다
- 나중에 봐도 어디까지 했는지 추적 가능하다

즉:

`빠르게 만들되, 지켜야 할 것만은 끝까지 지키는 것`

## 이 프로젝트의 권장 자기정의

이 프로젝트를 앞으로 설명할 때는 이렇게 정의하는 것이 가장 맞다.

### 짧은 정의

`검증형 바이브 코딩 프로젝트`

### 조금 긴 정의

`AI를 강하게 활용한 고속 바이브 코딩 프로젝트이지만, RHWP 엔진 경계와 BBDG 기능 보존만큼은 문서와 검증으로 통제하는 프로젝트`

### 가장 공식적인 정의

`RHWP 엔진 통합 보존 프레임워크는 바이브 코딩 성격이 강한 개발 프로젝트이며, 빠른 구현과 실사용 피드백 중심으로 진행하되, 엔진 경계·기능 보존·UI/UX 회귀·검증 게이트는 문서 기반으로 통제한다.`

## 최종 선언

이 프로젝트는 AI가 대신 생각하는 프로젝트도 아니고,
전통적인 무거운 문서 중심 개발 프로젝트도 아니다.

이 프로젝트는
`바이브 코딩으로 빠르게 전진하되,
무너지면 안 되는 경계만은 끝까지 붙잡는 프로젝트`
다.
