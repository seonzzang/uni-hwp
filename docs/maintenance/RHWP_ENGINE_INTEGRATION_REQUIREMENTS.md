# RHWP Engine Integration Requirements

Project:
- `RHWP Integration Preservation Framework`
- `RHWP 엔진 통합 보존 프레임워크`

## 목적

BBDG HWP Editor가 RHWP 엔진의 향후 업데이트를 충돌 없이 받아들이면서, BBDG 전용 UX/인쇄/파일 처리 기능을 안정적으로 유지할 수 있도록 엔진 통합 요구사항을 정의한다.

## 배경

RHWP는 HWP/HWPX 파싱, 문서 모델, 페이지 계산, SVG 렌더링, hitTest, export 등 핵심 문서 엔진 역할을 한다. RHWP 엔진은 아직 계속 개선될 가능성이 높으므로 BBDG HWP Editor가 RHWP 코어를 직접 수정하면 upstream 업데이트 병합 비용과 충돌 위험이 증가한다.

따라서 BBDG HWP Editor는 RHWP 코어를 제품 내부 구현물이 아니라 교체 가능한 엔진 의존성으로 취급해야 한다.

## 범위

포함:
- RHWP 엔진 업데이트 수용 정책
- BBDG 앱 레이어와 RHWP 엔진 경계 정의
- 엔진 API 변경 대응 요구사항
- 기존 BBDG 기능 보존 요구사항
- 검증 및 회귀 테스트 요구사항

제외:
- RHWP 엔진 자체의 렌더링 품질 개선 구현
- HWP/HWPX 포맷 파서 내부 알고리즘 변경
- upstream RHWP 릴리스 정책

## 핵심 원칙

0. 지금까지 BBDG HWP Editor에 구현한 기능과 UX는 RHWP 엔진 업데이트 이후에도 유지한다.
1. RHWP 코어는 가능한 한 수정하지 않는다.
2. RHWP upstream 엔진 업데이트는 확인 가능한 즉시 검토하고 반영 가능성을 평가한다.
3. BBDG 전용 기능은 앱, Tauri, worker, adapter 레이어에 구현한다.
4. 엔진 API 접근은 `wasm-bridge` 같은 단일 어댑터 경계를 통해 수행한다.
5. RHWP 업데이트 시 BBDG 기능이 자동 유지되도록 엔진 의존도를 낮춘다.
6. 엔진 수정이 필요하면 upstream RHWP에 반영 가능한 일반 기능으로 설계한다.
7. 모든 변경은 작은 단계로 나누고, 단계마다 에러/성능/기능/UI 회귀를 검증한다.
8. 모든 단계는 `RHWP_ENGINE_GUARDIAN_AGENT.md` 기준 guardian review를 통과해야 한다.
9. 다음 단계 진행은 오류 검증과 기능 유지 검증이 모두 통과된 경우에만 허용한다.
10. `RHWP_ENGINE_ORCHESTRATION_SUPERVISOR.md` 기준으로 작업 순서, 문서 확인, 단계 이동을 감독한다.
11. `RHWP_ENGINE_MOMENTUM_MONITOR.md` 기준으로 작업 정체와 다음 행동 누락을 감지한다.
12. `RHWP_ENGINE_BASELINE_COMPARISON_AGENT.md` 기준으로 기준 앱 대비 UI/UX와 기능 동등성을 확인한다.
13. `RHWP_ENGINE_APP_CONTROL_VERIFICATION_AGENT.md` 기준으로 가능한 범위에서 앱 직접 조작 검증을 수행한다.

## 계층 구분 요구사항

### RHWP 코어 영역

다음 영역은 upstream RHWP 소스로 취급한다.

- `src/**/*.rs`
- `src/wasm_api.rs`
- `pkg/rhwp.js`
- `pkg/rhwp_bg.wasm`
- `pkg/*.d.ts`

요구사항:
- BBDG 전용 기능을 이 영역에 직접 구현하지 않는다.
- 직접 수정이 필요한 경우 별도 사유, 대체 불가 판단, upstream 반영 가능성을 문서화한다.
- `pkg/*`는 수동 수정하지 않고 RHWP 빌드 산출물로만 취급한다.

### BBDG 앱 영역

다음 영역은 BBDG 제품 기능 구현 영역으로 취급한다.

- `rhwp-studio/src/**`
- `src-tauri/src/**`
- `scripts/**`
- BBDG 전용 문서 및 설정

요구사항:
- 인쇄 UX, PDF 생성, 링크 드롭, 임시 파일 관리, 진행률/취소/ETA, UI 개선은 이 영역에 구현한다.
- RHWP 엔진의 raw API를 여러 곳에서 직접 호출하지 않는다.
- 엔진 API 변화는 adapter 한 곳에서 흡수한다.

### 어댑터 영역

대표 파일:
- `rhwp-studio/src/core/wasm-bridge.ts`

요구사항:
- 앱이 사용하는 RHWP 기능은 가능하면 이 계층을 통해 노출한다.
- RHWP API 변경 시 앱 전체가 아니라 어댑터만 수정하도록 설계한다.
- BBDG 전용 보정 로직은 어댑터 바깥으로 둘 수 있는지 먼저 검토한다.
- 어댑터가 비대해질 경우 기능별 adapter module로 분리한다.

## 기능 보존 요구사항

RHWP 엔진 업데이트 후에도 다음 BBDG 기능은 유지되어야 한다.

- 문서 열기, 새 문서, 저장 흐름
- HWP/HWPX 렌더링 및 페이지 표시
- 대형 문서 canvas/page window 기반 표시 안정성
- 연속 문서 교체 및 링크 드롭 시 WASM 문서 교체 안정성
- 로드 전 입력/hitTest 경고 억제 및 null pointer 방지
- 웹폰트 지연 등록
- 실패 웹폰트 재시도 방지
- 브라우저 드래그 데이터 기반 원격 HWP/HWPX 링크 후보 추출
- URL 확장자 기반 원격 문서 판별
- 응답 헤더 기반 원격 문서 판별
- 비문서 다운로드 WASM 로드 전 차단
- 원격 문서 임시 다운로드 및 자동 정리
- 원격 HWP/HWPX 링크 드롭 성공/실패 로그와 안내
- `[파일] -> [인쇄]` 단일 메뉴 진입 구조
- 인쇄 대화창
- 인쇄 범위 선택
- 페이지 범위 입력창 클릭 시 `페이지 범위` 자동 선택
- 끝 페이지 입력창 Enter 실행
- `PDF 내보내기` 방식
- 대화창 내부 `인쇄` 방식
- 기존 브라우저 인쇄 경로
- PDF 내보내기 및 앱 내부 PDF 뷰어
- 대형 문서 PDF 생성 청크 처리
- PDF 병합 파이프라인
- PDF 생성 진행률 표시
- PDF 생성 경과 시간 표시
- PDF 생성 전체 남은 시간 표시
- PDF 생성 취소
- ETA 학습 기반 평균 시간 추정
- 앱 내부 PDF 뷰어
- PDF 뷰어에서 편집기로 복귀
- 불필요한 PDF 청크 미리보기 메뉴 제거 상태

이 요구사항은 엔진 교체성보다 우선한다. RHWP 코어를 깨끗하게 유지하더라도 위 기능이 회귀되면 업데이트 작업은 완료로 간주하지 않는다.

## UI/UX 보존 요구사항

RHWP 엔진 업데이트 후에도 사용자가 보는 BBDG HWP Editor의 화면 구조와 주요 사용 흐름은 최대한 유지되어야 한다.

보존 대상:
- 파일 메뉴 구조
- `[인쇄]` 메뉴 진입 흐름
- 인쇄 대화창 레이아웃
- 인쇄 범위 선택 UX
- 페이지 범위 입력창 자동 선택 동작
- 끝 페이지 입력창 Enter 실행 동작
- 인쇄 방식 선택 UX
- PDF 생성 진행 오버레이
- 진행률, 경과 시간, 전체 남은 시간 표시
- 취소 버튼 위치와 동작
- 앱 내부 PDF 뷰어 헤더 구조
- PDF 뷰어에서 편집기로 복귀 UX
- 링크 드롭 성공/실패 안내 흐름

동작은 유지되더라도 UI/UX 흐름이 달라져 사용자가 혼란을 느끼면 회귀로 간주한다.

## 엔진 업데이트 요구사항

RHWP upstream 업데이트를 적용할 때 다음 순서를 따른다.

1. RHWP upstream 변경사항은 확인 가능한 즉시 fetch/확인 대상에 올린다.
2. 변경 감지 후 가능한 빠르게 영향 범위를 검토한다.
3. RHWP upstream 변경사항을 별도 브랜치에서 가져온다.
4. `src`, `pkg`, `Cargo.toml`, generated bindings 변화를 확인한다.
5. BBDG 앱 레이어 변경과 충돌 여부를 확인한다.
6. `wasm-bridge`에서 깨진 API를 흡수한다.
7. 앱 기능 회귀 테스트를 수행한다.
8. 엔진 내부 패치가 필요한 경우 별도 패치 파일 또는 upstream PR 후보로 분리한다.

추가 요구사항:

- upstream 변경을 장기간 방치하지 않는다.
- 즉시 반영이 어렵다면 blocker, 영향 범위, 보류 사유를 문서화한다.
- “나중에 몰아서 업데이트”를 기본 전략으로 삼지 않는다.
- RHWP 엔진 업데이트 확인은 선택 작업이 아니라 상시 유지 작업으로 취급한다.

## 금지 요구사항

- BBDG 전용 UI/UX를 `src/wasm_api.rs`에 직접 구현하지 않는다.
- PDF 생성, 인쇄 진행률, 링크 드롭 처리를 RHWP 코어에 넣지 않는다.
- `pkg/rhwp_bg.wasm`을 직접 수정하거나 수동 보정하지 않는다.
- upstream 업데이트와 BBDG 기능 변경을 하나의 커밋에 섞지 않는다.
- 엔진 교체 검증 없이 `pkg/*`만 갱신하지 않는다.

## 예외 요구사항

엔진 코어 수정이 허용되는 경우:
- RHWP 자체 버그 수정
- HWP/HWPX 표준 해석 개선
- 렌더링 정확도 개선
- 공개 API가 필요한 일반 기능 추가
- upstream에 그대로 제안 가능한 변경

예외 조건:
- 변경 이유를 문서화해야 한다.
- BBDG 전용 의존성이 없어야 한다.
- 최소 재현 파일 또는 테스트가 있어야 한다.
- 가능하면 upstream PR 형태로 유지한다.

## 검증 요구사항

기본 검증:
- `cargo check`
- `cargo test` 가능한 범위
- `npm run build`
- 앱 실행 및 문서 로드

수동 회귀 검증:
- 로컬 HWP 열기
- 로컬 HWPX 열기
- 원격 HWP/HWPX 링크 드롭
- 1쪽 문서 표시
- 50쪽 이상 문서 표시
- 300쪽 이상 문서 PDF 내보내기
- 전체 문서 PDF 내보내기
- PDF 생성 취소
- 앱 내부 PDF 뷰어에서 편집기로 복귀

성능 검증:
- 대형 문서 로딩 시간
- 첫 페이지 표시 시간
- 스크롤 반응성
- PDF 데이터 준비 시간
- PDF 생성 시간
- PDF 병합 시간
- PDF 저장 시간
- 메모리 증가 추세
- WASM panic 또는 null pointer 발생 여부

진행 원칙:
- 한 단계에 한 가지 목적만 수행한다.
- 각 단계마다 검증 후 커밋한다.
- 오류 검증과 기능 유지 검증이 모두 통과해야 다음 단계로 진행한다.
- 실패 또는 성능 급락이 발견되면 다음 단계로 진행하지 않는다.
- UI/UX 변경이 불가피한 경우 별도 사유를 문서화한다.
- guardian review가 `Stop`이면 다음 단계로 진행하지 않는다.
- 작업이 멈췄거나 다음 행동이 불명확하면 momentum monitor 기준으로 다음 최소 행동을 정한다.
- 업데이트 수용 전 기준 앱과 변경 앱의 UI/UX 및 기능 동등성을 비교한다.
- 직접 자동화 가능한 UI/UX 검증은 앱 조작, 스크린샷, 로그 확인으로 수행한다.
- 자동화가 어려운 OS-native 동작은 사용자 보조 확인으로 명시한다.

## 산출물 요구사항

엔진 업데이트 작업마다 다음 산출물을 남긴다.

- 업데이트 대상 RHWP 버전 또는 커밋
- 충돌 파일 목록
- 어댑터 수정 목록
- 엔진 내부 수정 여부
- 회귀 테스트 결과
- guardian review 결과
- baseline comparison 결과
- 남은 위험 및 후속 작업
