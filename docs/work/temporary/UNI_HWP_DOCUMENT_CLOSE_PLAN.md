# Uni-HWP 문서 닫기 및 저장 확인 작업 계획 명세서

문서 버전:
- `8.1.101`

문서 성격:
- 내부 작업 문서
- 작업 완료 후 `docs/work/archive`로 이동하거나 삭제 가능

참조 요구사항:
- `docs/work/temporary/UNI_HWP_DOCUMENT_CLOSE_REQUIREMENTS.md`

상위 원칙:
- RHWP 엔진 코어는 수정하지 않는다.
- 기능 구현은 Uni-HWP App Shell Layer에서 해결한다.
- 각 단계는 `오류 없음`과 `기능 유지` 두 조건이 확인된 뒤 다음 단계로 진행한다.

## 1. 작업 목적

본 계획의 목적은 다음 기능을 안전하게 구현하는 것이다.

1. 현재 문서를 닫는 UI와 동작 추가
2. 변경 문서에 대한 저장 확인 팝업 추가
3. 앱 종료 시에도 동일한 보호 흐름 적용

## 2. 구현 전략

핵심 전략은 다음과 같다.

- RHWP 엔진에 “문서 닫기” 개념을 추가하지 않는다.
- 앱 셸에서 현재 문서의 존재, 파일명, dirty 상태, 닫기 의도(close intent)를 관리한다.
- 저장 확인은 공통 서비스/헬퍼로 모아 `문서 닫기`와 `앱 종료`에서 재사용한다.
- 문서 닫기는 “현재 문서 편집 세션을 종료하고 빈 상태로 전환”하는 앱 라이프사이클 작업으로 구현한다.

## 3. 단계별 작업 계획

### 단계 1: 현 상태 분석 및 닫기 진입점 정의

목표:
- 현재 문서 라이프사이클 진입점과 종료 가능한 지점을 식별한다.

작업:
- `bootstrap-editor.ts`에서 편집기 초기화와 UI 연결 구조 확인
- `command/commands/file.ts`에서 파일 열기/저장/새로 만들기 흐름 확인
- `menu-bar.ts`와 관련 CSS에서 상단 우측 버튼 삽입 위치 확인
- 앱 종료 이벤트를 어디에서 연결할지 확인

완료 기준:
- 문서 닫기 버튼 삽입 위치와 종료 훅 연결 위치가 확정된다.
- 엔진 코어 수정 없이 구현 가능한 경로가 명확해진다.

### 단계 2: 문서 편집 상태 모델 추가

목표:
- 현재 문서의 dirty 상태와 표시용 파일명 상태를 앱 셸에서 추적할 수 있게 한다.

작업:
- 문서 상태 모델 또는 파일 명령 계층에 최소 상태 필드 정의
  - 문서 열림 여부
  - 파일명/표시명
  - 저장 경로 존재 여부
  - dirty 여부
- 새 문서/열기/저장/저장 완료/문서 닫기 시 상태 전이 정의

완료 기준:
- dirty 상태가 일관되게 읽히고 갱신된다.

### 단계 3: 상단 우측 문서 닫기 UI 추가

목표:
- 편집기 상단 우측에 문서 닫기 버튼을 추가한다.

작업:
- 메뉴바 DOM 또는 상단 레이아웃에 `문서 닫기` 컨트롤 삽입
- 운영체제 창 닫기 `X`와 혼동되지 않도록 스타일 조정
- 문서 없음 상태일 때 숨김 또는 비활성 처리

완료 기준:
- 문서 열림 상태에서만 버튼이 자연스럽게 표시된다.

### 단계 4: 문서 닫기 동작 구현

목표:
- 현재 문서를 안전하게 닫고 편집기를 빈 상태로 되돌린다.

작업:
- `closeCurrentDocument()` 성격의 앱 셸 동작 추가
- 입력 핸들러, 캔버스 뷰, 선택 상태, 페이지 표시 상태를 초기화
- 기존 문서가 해제된 뒤 편집기가 다시 새 문서/열기 명령을 받을 수 있게 보장

완료 기준:
- 문서 닫기 후 빈 상태 편집기로 전환된다.

### 단계 5: 저장 확인 팝업 공통화

목표:
- 닫기와 종료에서 재사용할 저장 확인 공통 흐름을 만든다.

작업:
- `파일명 문서를 저장하겠습니까?` 팝업 문구 정의
- 선택지:
  - 저장
  - 저장 안 함
  - 취소
- 기존 저장/다른 이름으로 저장 로직과 연결
- 새 문서처럼 파일명이 없는 경우 대체 표시명 적용

완료 기준:
- 공통 헬퍼 하나로 닫기/종료에 동일 정책이 적용된다.

### 단계 6: 앱 종료 훅 연동

목표:
- 앱 종료 시에도 dirty 상태가 있으면 저장 여부를 묻도록 한다.

작업:
- Tauri/브라우저 창 종료 시점 훅 연결
- 문서 닫기와 동일한 저장 확인 흐름 재사용
- 이중 팝업 방지

완료 기준:
- 종료 시 저장 확인이 정확히 한 번만 뜬다.

### 단계 7: 회귀 검증 및 UX 다듬기

목표:
- 기존 기능을 유지하면서 새 UX를 마감한다.

작업:
- 파일 열기/새로 만들기/저장/저장 안 함/닫기/종료 흐름 수동 점검
- 버튼 색상, 명도, hover, disabled 상태 다듬기
- 필요 시 문구를 사용자 친화적으로 조정

완료 기준:
- 기능과 UX가 모두 안정적이다.

## 4. 권장 파일 분리 방향

작업 중 코드가 엉키지 않도록 다음 분리를 우선한다.

- UI 표시:
  - 메뉴바 버튼 생성/상태 반영
- 상태 관리:
  - 현재 문서 세션 상태
- 명령 처리:
  - 닫기 요청
  - 종료 요청
- 공통 대화:
  - 저장 확인 팝업

필요 시 신규 파일 후보:
- `rhwp-studio/src/app/document-session.ts`
- `rhwp-studio/src/app/document-close-guard.ts`

단, 실제 코드베이스 구조를 보고 과도한 분리는 피한다.

## 5. 위험 요소 및 대응

### 위험 1
- dirty 상태가 실제 수정과 어긋날 수 있음

대응:
- 저장/열기/새로 만들기/닫기 전이 지점을 먼저 명확히 모델링한다.

### 위험 2
- 앱 종료와 문서 닫기에서 저장 확인 로직이 복제될 수 있음

대응:
- 공통 헬퍼를 먼저 설계한다.

### 위험 3
- 문서 닫기 후 엔진/캔버스 잔여 상태가 남을 수 있음

대응:
- 편집 세션 종료 순서를 명시적으로 정의하고 검증한다.

### 위험 4
- 운영체제 창 닫기 `X`와 문서 닫기 `X`가 혼동될 수 있음

대응:
- 위치, 색, 크기, hover, tooltip로 역할을 분리한다.

## 6. 검증 시나리오

필수 시나리오:

1. 문서 열기 후 수정 없이 닫기
2. 문서 열기 후 수정 후 닫기 -> 저장
3. 문서 열기 후 수정 후 닫기 -> 저장 안 함
4. 문서 열기 후 수정 후 닫기 -> 취소
5. 새 문서 생성 후 수정 후 앱 종료
6. 기존 파일 열기 후 수정 후 앱 종료
7. 문서 닫기 후 새 파일 다시 열기
8. 문서 닫기 후 URL 드롭으로 다시 열기

## 7. 성공 기준

- RHWP 엔진 코어 무수정
- 문서 닫기 버튼 정상 표시
- dirty 문서 보호 팝업 정상 동작
- 종료와 닫기 흐름에서 공통 로직 재사용
- 기존 파일 열기/저장/인쇄/PDF/드래그 앤 드롭 기능 비회귀
