# Uni-HWP 문서 닫기 및 저장 확인 요구사항 명세서

문서 버전:
- `8.1.101`

문서 성격:
- 내부 작업 문서
- 작업 완료 후 `docs/work/archive`로 이동하거나 삭제 가능

문서 목적:
- Uni-HWP에 `문서 닫기(close)` UX와 `저장 여부 확인` UX를 추가하기 위한 요구사항을 정의한다.
- RHWP 엔진 코어를 보존하면서, Uni-HWP 앱 셸 계층만 수정하는 것을 전제로 한다.

관련 상위 원칙:
- `README.md`의 `Embedded RHWP Engine Layer`, `Black-Box Adapter Layer`, `Uni-HWP App Shell Layer` 구조를 따른다.
- `docs/public/architecture/RHWP_INTEGRATION_PRESERVATION_ARCHITECTURE.md`의 변경 허용 경계를 따른다.
- RHWP 엔진 코어는 가능한 한 직접 건드리지 않는다.
- 문서 닫기/저장 확인은 앱 셸 UX와 문서 라이프사이클 계층에서 해결한다.

## 1. 배경

현재 Uni-HWP는 문서를 열고 편집하는 흐름은 제공하지만, 사용자가 앱 내부에서 현재 문서를 명시적으로 닫는 UX는 제공하지 않는다.

또한 문서에 변경이 발생한 상태에서:
- 앱을 종료하려는 경우
- 현재 문서를 닫으려는 경우

저장 여부를 묻는 일관된 보호 UX가 필요하다.

이 기능은 엔진 렌더링이나 파싱 기능이 아니라, 문서 상태 추적과 사용자 보호를 담당하는 앱 셸 기능이다.

## 2. 목표

본 작업의 목표는 다음 두 가지다.

1. 사용자가 현재 열려 있는 문서를 앱 내부에서 명시적으로 닫을 수 있어야 한다.
2. 변경된 문서를 저장하지 않고 닫거나 종료하는 실수를 방지해야 한다.

## 3. 대전제

### 3.1 아키텍처 원칙

- RHWP 엔진 레이어는 수정하지 않는다.
- WASM 파서/렌더러/문서 모델/코어 명령 계층에는 기능을 추가하지 않는다.
- 기능 구현은 Uni-HWP App Shell Layer와 필요한 경우 Black-Box Adapter Layer의 문서 라이프사이클 보호 범위 안에서만 수행한다.

### 3.2 구현 경계

수정 허용 범위:
- `rhwp-studio/src/app/**`
- `rhwp-studio/src/ui/**`
- `rhwp-studio/src/command/**`
- `rhwp-studio/src/styles/**`
- 필요 시 `rhwp-studio/src/core/wasm-bridge.ts`의 라이프사이클 보호 범위 안의 얇은 보조 상태 연동

수정 금지 범위:
- RHWP Rust 코어
- 파서/레이아웃/렌더링 로직
- WASM 엔진 내부 명령 구조

## 4. 기능 요구사항

## 4.1 문서 닫기 버튼 추가

### FR-001
- 편집기 상단 메뉴바 라인의 우측 끝에 현재 문서를 닫는 버튼을 추가해야 한다.

### FR-002
- 이 버튼은 앱 종료용 창 제목 표시줄의 `X`와 시각적으로 혼동되지 않도록 배치와 명도 차이를 가져야 한다.

### FR-003
- 닫기 버튼은 앱의 종료 버튼 바로 아래 시각 축에 위치하되, 운영체제 창 제어와 기능적으로 혼동되지 않아야 한다.

### FR-004
- 닫기 버튼을 클릭하면 현재 열려 있는 문서를 닫는 흐름이 시작되어야 한다.

### FR-005
- 현재 문서가 없는 상태에서는 닫기 버튼이 비활성화되거나 숨김 상태여야 한다.

## 4.2 문서 닫기 동작

### FR-006
- 저장되지 않은 변경이 없는 문서는 확인 절차 없이 즉시 닫을 수 있어야 한다.

### FR-007
- 문서를 닫은 뒤 편집기는 문서 미열림 상태로 안전하게 전환되어야 한다.

### FR-008
- 문서 닫기 후 기존 문서의 입력 상태, 커서 상태, 선택 상태, 페이지 표시 상태가 남지 않아야 한다.

### FR-009
- 문서 닫기 후 새 문서 열기, 새로 만들기, 드래그 앤 드롭 열기 등 기존 진입 흐름은 그대로 정상 작동해야 한다.

## 4.3 변경 감지 및 저장 확인

### FR-010
- 현재 문서에 변경이 발생했는지 추적할 수 있어야 한다.

### FR-011
- 사용자가 앱 종료를 시도할 때 현재 문서가 변경된 상태라면 저장 여부 확인 팝업을 띄워야 한다.

### FR-012
- 사용자가 문서 닫기를 시도할 때 현재 문서가 변경된 상태라면 저장 여부 확인 팝업을 띄워야 한다.

### FR-013
- 팝업 문구는 다음 의미를 만족해야 한다.
  - `'<파일명>' 문서를 저장하겠습니까?`

### FR-014
- 파일명이 없는 새 문서 또는 경로 미지정 문서는 사용자에게 자연스러운 대체 표기를 사용해야 한다.
  - 예: `새 문서`

### FR-015
- 팝업은 최소 다음 선택지를 제공해야 한다.
  - 저장
  - 저장 안 함
  - 취소

### FR-016
- `저장` 선택 시:
  - 저장 가능한 문서는 저장 후 닫기/종료를 이어가야 한다.
  - 저장 경로가 없는 문서는 기존 저장/다른 이름으로 저장 흐름과 연결되어야 한다.

### FR-017
- `저장 안 함` 선택 시:
  - 변경사항을 폐기하고 닫기/종료를 계속해야 한다.

### FR-018
- `취소` 선택 시:
  - 닫기/종료 동작을 취소하고 현재 편집 상태를 유지해야 한다.

## 4.4 앱 종료 연동

### FR-019
- 운영체제 창 닫기 버튼 또는 앱 종료 명령으로 종료할 때에도 동일한 저장 확인 규칙이 적용되어야 한다.

### FR-020
- 문서가 저장되지 않은 상태가 아니면 종료는 기존처럼 즉시 진행되어야 한다.

## 5. 비기능 요구사항

### NFR-001
- 새로 추가되는 팝업은 기존 Uni-HWP 다이얼로그 스타일과 시각적으로 일관되어야 한다.

### NFR-002
- 문서 닫기 버튼은 “앱 닫기”가 아니라 “문서 닫기”라는 의미가 명확해야 한다.

### NFR-003
- 사용자가 오작동으로 느끼지 않도록, 닫기/저장 확인 흐름은 응답성이 좋아야 한다.

### NFR-004
- 문서 닫기 동작은 RHWP 엔진 내부 상태를 오염시키지 않아야 한다.

### NFR-005
- 문서 닫기 후 다른 문서를 다시 열 때 이전 문서의 잔여 상태가 남지 않아야 한다.

### NFR-006
- 앱 종료 훅과 문서 닫기 훅이 중복 실행되어 이중 팝업이 뜨면 안 된다.

### NFR-007
- dirty 상태 추적 로직은 메뉴/입력/UI 코드 전역에 흩어지지 않고, 문서 상태 관리 또는 파일 명령 계층에 집중되어야 한다.

### NFR-008
- 저장 확인 로직은 앱 종료와 문서 닫기에서 공통 재사용되어야 한다.

## 6. 비목표

- RHWP 엔진 편집 명령 체계를 재설계하지 않는다.
- 다중 탭/다중 문서 동시 편집은 이번 범위에 포함하지 않는다.
- 자동 저장 기능은 이번 범위에 포함하지 않는다.
- 문서 비교/복구 기능은 이번 범위에 포함하지 않는다.

## 7. 수용 기준

### AC-001
- 문서가 열려 있을 때 상단 우측에 문서 닫기 버튼이 보인다.

### AC-002
- 문서가 없을 때는 해당 버튼이 보이지 않거나 비활성화된다.

### AC-003
- 변경 없는 문서는 클릭 즉시 닫힌다.

### AC-004
- 변경 있는 문서는 닫기 시 저장 확인 팝업이 뜬다.

### AC-005
- 앱 종료 시에도 같은 저장 확인 팝업이 뜬다.

### AC-006
- `저장`, `저장 안 함`, `취소` 동작이 각각 올바르게 연결된다.

### AC-007
- 문서 닫기 후 새 문서 열기/새로 만들기/드래그 앤 드롭이 기존처럼 정상 동작한다.

### AC-008
- RHWP 엔진 코어 파일은 수정되지 않는다.
