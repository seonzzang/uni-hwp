# Branch Document Policy

이 문서는 Uni-HWP 내부 개발자를 위한 문서 운영 정책이다.

목적:
- 작업 브랜치에서 생성되는 문서와 릴리스 브랜치에 반영되는 문서를 명확히 분리한다.
- 외부 공개 문서와 내부 작업 문서가 섞이지 않도록 기준을 고정한다.

## 기본 원칙

1. 업그레이드 작업, 기능 개발, 실험은 별도의 작업 브랜치에서 수행한다.
2. 작업 중 생성되는 문서는 `docs/public`과 `docs/work`로 구분한다.
3. 릴리스 브랜치에는 원칙적으로 `docs/public`만 반영한다.
4. `docs/work` 아래 문서는 내부 개발/기록용이며 외부 배포 대상으로 간주하지 않는다.

## 브랜치 역할

### 작업 브랜치

예:
- `upgrade`
- 기능별 실험 브랜치
- 리팩토링 브랜치

허용 문서:
- `docs/public/**`
- `docs/work/temporary/**`
- `docs/work/archive/**`

설명:
- 작업 브랜치에서는 공개 문서와 내부 작업 문서를 모두 사용할 수 있다.
- 요구사항, 계획서, 검증 메모, 실험 기록은 우선 작업 브랜치에서 작성한다.

### 릴리스 브랜치

예:
- 외부 배포 기준 브랜치

허용 문서:
- `docs/public/**`

원칙:
- `docs/work/**`는 릴리스 브랜치로 가져가지 않는다.
- 릴리스 브랜치에는 외부 공개가 가능한 문서만 유지한다.
- 외부 저장소/배포본으로 내보낼 때도 동일 원칙을 따른다.

## 문서 분류 규칙

### `docs/public`

성격:
- 외부 공개 가능
- 장기 유지 문서
- 제품, 아키텍처, 유지보수, 릴리스, 법무 문서

릴리스 반영:
- 반영 대상

### `docs/work/temporary`

성격:
- 작업 중 임시 문서
- 완료 후 삭제 가능

릴리스 반영:
- 반영 금지

### `docs/work/archive`

성격:
- 내부 장기 보관용 작업 문서
- 다시 참고할 가치가 있는 계획/기록/결정 문서

릴리스 반영:
- 반영 금지

## 운영 절차

1. 작업 브랜치에서 기능 개발과 함께 필요한 문서를 작성한다.
2. 공개 가치가 있는 문서는 `docs/public`에 둔다.
3. 임시 메모/계획/검증 문서는 `docs/work/temporary` 또는 `docs/work/archive`에 둔다.
4. 릴리스 브랜치로 반영할 때는 코드와 함께 `docs/public`만 선별 반영한다.
5. 배포 전에 `docs/work` 문서가 섞이지 않았는지 확인한다.

## 금지 사항

- 내부 작업 문서를 무심코 릴리스 브랜치에 함께 올리지 않는다.
- 공개 문서와 내부 문서를 같은 폴더에 섞어 저장하지 않는다.
- 루트 폴더에 작업 메모성 문서를 새로 만들지 않는다.

## 한 줄 정책

작업 브랜치에서는 `docs/public`과 `docs/work`를 모두 사용하고, 릴리스 브랜치에는 `docs/public`만 반영한다.
